# Как работать с репозиторием (для команды)
Краткие правила: ветки, пуши, релизы, подключение разработчиков.

---

## Ветки и основной код

- **`main`** — основная ветка. В ней всегда стабильная, рабочая версия.
- Работать можно так:
  - **Простой вариант:** все коммитят и пушат в `main` (подходит для маленькой команды).
  - **Вариант с ветками:** для каждой задачи создаётся ветка (например `feature/название` или `fix/баг`), потом изменения вливаются в `main` через Pull Request (PR).

---

## Обычный цикл: правка → пуш

1. **Скачать последние изменения** (если уже не один работаешь):
   ```bash
   git pull origin main
   ```
2. **Внести правки** в файлы.
3. **Добавить изменения и закоммитить:**
   ```bash
   git add .
   git commit -m "Краткое описание: что сделано"
   ```
4. **Отправить на GitHub:**
   ```bash
   git push origin main
   ```

Если работаешь в своей ветке — вместо `main` подставляешь имя ветки и потом делаешь PR в `main` на GitHub.

---

## Релизы (версии проекта)

Релиз — это «зафиксированная» версия (например v1.0.0), чтобы можно было ссылаться на неё и скачивать архив.

**Способ 1 — тег в Git (минимум):**

```bash
# После коммитов на main:
git tag v1.0.0
git push origin v1.0.0
```

**Способ 2 — Release на GitHub (как делают чаще всего):**

1. На GitHub: репозиторий → вкладка **Releases** → **Create a new release**.
2. **Choose a tag:** создать тег, например `v1.0.0`.
3. Заполнить заголовок и описание (что вошло в релиз).
4. Опубликовать (**Publish release**).

Так у проекта появятся версии, которые видны в разделе Releases и которые могут скачать другие.

---

## Подключение других программистов

1. **Владелец репозитория** заходит на GitHub → репозиторий **RASHOTKLIK** → **Settings** → слева **Collaborators** (или **Collaborators and teams**).
2. **Add people** → ввести логин GitHub или email приглашаемого.
3. Приглашённый получает письмо/уведомление, принимает приглашение.
4. После этого он может:
   - **Клонировать репозиторий:**  
     `git clone https://github.com/ЛОГИН_ВЛАДЕЛЬЦА/RASHOTKLIK.git`
   - Делать `git pull`, создавать ветки, коммитить и пушить (в зависимости от прав).

Пароль при push не подойдёт — нужен **Personal Access Token** (GitHub → Settings → Developer settings → Personal access tokens). Токен вводится вместо пароля при первом push.

---

## Кратко

| Действие              | Команда / место                          |
|-----------------------|------------------------------------------|
| Взять последние изменения | `git pull origin main`               |
| Закоммитить и отправить   | `git add .` → `git commit -m "..."` → `git push origin main` |
| Создать релиз (тег)       | `git tag v1.0.0` → `git push origin v1.0.0` или через Releases на GitHub |
| Добавить разработчика     | GitHub → Settings → Collaborators → Add people |

Если договоритесь о ветках и PR — можно дописать сюда отдельный пункт под ваш workflow.
